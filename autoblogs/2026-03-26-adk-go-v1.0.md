# ADK Go v1.0.0 — Go 言語エージェント開発キットが安定版へ到達した設計的意味

2026-03-26 時点で、ADK Go が v0.6.0 から v1.0.0 安定版へと到達した。Python の v1.27.4 や JS の v0.2.4 と並走する 3 言語エコシステムの中で、Go 版が「安定版」を最初に宣言した意味は何か。v0.5.0・v0.6.0 の積み重ねと、v1.0.0 以降に積み上がりつつある未リリース変更群を通じて、**ADK Go の設計思想**と今後の方向性を整理する。

## v0.x 系の積み重ね——v1.0.0 への布石

### v0.5.0：可観測性とプラグインアーキテクチャの確立

v0.5.0（2026-03 以前リリース）は、単なる機能追加ではなく **ADK Go のアーキテクチャ的な転換点** だった。

| 変更 | 詳細 | 意味 |
|------|------|------|
| OpenTelemetry Semconv Tracing | セマンティック規約に基づくトレーシング完全実装 | Python ADK との可観測性統一 |
| Debug Endpoints v2 | GCP quota/resource project バリデーション付き | エンタープライズ運用への対応 |
| Retry and Reflect プラグイン | リトライ・リフレクションプラグイン追加 | 自律エージェント耐障害性の実装 |
| Logging プラグイン | ロギングプラグイン追加 | 運用診断ツールの充実 |
| Preload Memory ツール | メモリプリロードサポート | 長時間稼働エージェントの状態管理 |

**OpenTelemetry Semconv Tracing** の完全実装は、Python ADK v1.23.0 で BigQuery プラグイントレーシング移行が行われたのと同時期に並走している。これは単一言語の実装ではなく、**ADK エコシステム全体の可観測性統一**という設計方針を示している。

### v0.6.0：拡張性の整備

v0.6.0 では **RemoteAgent パーツ変換の拡張ポイント** (#627) が導入された。これは A2A プロトコル（v1.0.0 正式リリース、2026-03-12）との連携で重要な役割を果たす。A2A v1.0.0 では `SendMessageConfiguration.blocking` → `return_immediately` (#1507) というフィールドリネームがあり、RemoteAgent がこの変更に追従できる拡張性が必要だった。

v0.6.0 の主要変更を整理する：

| 変更 | PR/Issue | 詳細 |
|------|---------|------|
| Apigee モデルサポート | #639 | Apigee LLM モデル追加 |
| RemoteAgent 拡張ポイント | #627 | パーツ変換のカスタマイズ |
| pathPrefix 設定 | #616 | API ランチャーの設定項目追加 |
| Runner WithStateDelta | #615 | 状態差分オプション追加 |
| ExampleTool ファクトリ | #612 | 設定可能なファクトリ追加 |
| Replay プラグイン | #618 | リプレイ機能追加 |
| GitHub Actions Node 24 互換性 | #642 | CI インフラ更新 |

**Replay プラグイン** (#618) は特に注目に値する。エージェントのリプレイ能力は、デバッグと本番障害解析の両方で強力なツールになる。Python の v1.27.0 で導入された **Durable Runtime**（クラッシュからの永続ランタイム復帰）と組み合わせると、Go は「再現可能なエージェント実行」という設計思想を明確に示している。

## v1.0.0 の意味——安定 API の宣言

v1.0.0 は機能的にはシンプルな「正式リリース」だが、その **設計的意味** は大きい。

### セマンティックバージョニングとの関係

Go エコシステムにおける v1.0.0 は単なるバージョン番号ではない。Go モジュールシステムでは v1.0.0 到達後の後方非互換変更は `v2/` パスを要求する慣習があり、**API 安定性の宣言** を意味する。

Python ADK は v1.24.0 で `credential_manager` の引数変更（`callback_context` → `tool_context`）を行い、v1.23.0 で OpenTelemetry BigQuery プラグイントレーシング移行を実施している。これらの破壊的変更が v1.x の中で行われた。Go が v1.0.0 として宣言した今、同様の破壊的変更は **v2.0.0 系への移行** を意味することになる。

この対比は重要だ。Python ADK では v2.0.0a1（2026-03-24 以前にタグ付け）がすでに存在する。Go は v1.0.0 で「安定期に入った」と宣言したばかりだ。

### 機能カバレッジの現実

v1.0.0 時点での 3 言語の機能カバレッジを比較する：

| 機能 | Python v1.27.4 | Go v1.0.0 | JS v0.2.4 |
|------|---------------|-----------|-----------|
| Agent Registry | yes | - | - |
| Skills 仕様準拠 | yes | - | - |
| A2A 統合 | yes | yes | yes |
| Durable Runtime | yes | - | - |
| adk optimize | yes | - | - |
| MCP App UI Widgets | yes | - | - |
| OpenTelemetry | yes | yes | - |
| ストリーミング | yes | yes | yes |
| Replay プラグイン | - | yes | - |

Go の機能セットは Python と比較して限定的だが、これは **意図的な設計選択** だ。Go が担うのは、**パフォーマンスクリティカルで可観測性が重要なバックエンドエージェント**のユースケースに絞られている。

## v1.0.0 以降の未リリース変更——次の進化方向

v1.0.0 リリース後も開発は継続しており、以下の変更が積み上がっている。

### Progressive SSE ストリーミングと並列関数実行

**Progressive SSE ストリーミング** は ADK Go の次期主要機能だ。SSE（Server-Sent Events）によるストリーミングは Python ADK でも実装されているが、Go 版では **並列関数実行** と組み合わせたプログレッシブ実装が特徴的だ。

AG-UI v0.0.48（2026-03-24 リリース）で追加された **並列 LRO ツールコール**（ADK での並列 Long-Running Operations ツールコールハンドリング）との設計的な整合性が見える。複数の長時間実行ツールを並列で走らせながら、SSE でその進捗をストリーミングする——これは **エージェント UI の体験品質** に直結する。

### AutoCreateSession

**AutoCreateSession** は Runner での自動セッション作成機能だ。これは AG-UI v0.0.48 の **セッションキャッシュ改善**（(thread_id, user_id) によるキーイングでクロスユーザー衝突防止）との設計的な対応関係にある。

セッション管理は分散エージェントシステムの複雑さの源泉だ。AutoCreateSession は「セッションを明示的に作成しなくても動く」という開発者体験の改善であり、**エージェント開発の進入障壁低下** という方向性を示している。

### RemoteAgent/Executor サブタスククリーンアップ

A2A v1.0.0（2026-03-12）の正式リリースに伴い、RemoteAgent と Executor のサブタスク管理が改善されている。A2A v1.0.0 では Push Notification Config 統合 (#1500)、重複 ID 削除 (#1487)、`ListTaskPushNotificationConfigs` 複数形化 (#1486) といった **タスク管理の構造変更** が入っており、Go 側でのクリーンアップはこれへの追従だ。

### ストリーミング応答集約修正

エラー詳細と finish reason の保持修正は、プロダクション運用での信頼性に関わる。ストリーミングレスポンスの途中でエラーが発生した場合、その詳細が正しく保持・伝播されなければデバッグが困難になる。**成熟した本番運用に耐えるエラーハンドリング** という設計思想の現れだ。

## ADK Go の設計思想——Python との比較で見える方向性

### Python ADK との機能差の意味

Python ADK v1.27.3 の **任意モジュールインポート防止**（セキュリティ強化）、v1.27.4 の **LiteLLM 脆弱バージョン除外** といったセキュリティ変更は、Go には現時点で対応する変更がない。これは Go のセキュリティモデルが言語レベルで異なるためだ。Go は静的型付け・コンパイル時依存解決のため、**Python 固有の動的インポート攻撃ベクター** は存在しない。

この非対称性は重要な示唆を与える。Python ADK のセキュリティ強化は「Python の動的な性質への対処」であり、Go は言語特性上そのリスクを持たない。**言語固有のリスクに応じたセキュリティ強化** という分業が成立している。

### Progressive SSE とアーキテクチャの整合性

```
ADK Go v1.0.0+ (Progressive SSE):

Agent
  |
  +-- Runner (AutoCreateSession)
        |
        +-- Tool Call 1 (parallel)--+
        +-- Tool Call 2 (parallel)--+-- SSE Stream --> Client
        +-- Tool Call 3 (parallel)--+
```

Legend / 凡例:
- Runner: セッション管理とエージェント実行を担うコアコンポーネント
- AutoCreateSession: 明示的なセッション作成なしに動作する機能
- parallel: 並列実行される複数ツールコール
- SSE Stream: Server-Sent Events による進捗ストリーミング

この設計は AG-UI の **並列 LRO ツールコール** と **Progressive SSE** の組み合わせに直接対応する。バックエンド（ADK Go）とフロントエンドプロトコル（AG-UI）が設計レベルで整合している。

## v1.0.0 → v2.0.0 への移行展望

Python ADK の v2.0.0a1 が示すように、ADK エコシステムは v2 系への移行を見据えている。Go は v1.0.0 に到達したばかりだが、以下の方向性が示唆される：

- **Skills ファースト設計**: Python で先行する Skills（スキル仕様準拠、GCS Skills）の Go 実装
- **Agent Registry**: エージェントの登録・発見機能の追加
- **Durable Runtime**: クラッシュ耐性のある永続ランタイム（Python で先行）
- **MCP App UI Widgets**: UI インテグレーションの拡張

Progressive SSE と AutoCreateSession は v1.x 系の次バージョン（v1.1.0 相当）として入る見込みが高く、v2.0.0 では設計思想の刷新が行われるだろう。

## まとめ

ADK Go v1.0.0 の到達は、機能的なマイルストーンではなく **API 安定性の宣言** だ。v0.5.0 での OpenTelemetry 完全実装と Retry/Reflect プラグイン、v0.6.0 での RemoteAgent 拡張ポイントと Replay プラグインを経て、v1.0.0 は「Go エージェント開発の基盤が固まった」ことを示している。

v1.0.0 以降の Progressive SSE ストリーミング・AutoCreateSession・RemoteAgent クリーンアップは、**高並列・高信頼性のバックエンドエージェント基盤** としての ADK Go の進化方向を明確にする。Python（v1.27.4 → v2.0.0a1）、JS（v0.2.4）との 3 言語エコシステムの中で、Go は独自のポジションを確立しつつある。

A2A v1.0.0（2026-03-12）の安定化、AG-UI v0.0.48（2026-03-24）の並列ツールコール対応、そして ADK Go v1.0.0——これらが揃うことで、**エンタープライズグレードのマルチエージェントシステム** の実装が現実的になってきた。
