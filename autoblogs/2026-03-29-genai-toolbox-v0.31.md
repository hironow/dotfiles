# GenAI Toolbox v0.31.0 — Protected Resource Metadata と enable-api フラグが拓く認可モデルの新局面

v0.30.0 の「One Skill per Toolset」設計転換からわずか 6 日後、GenAI Toolbox v0.31.0 (2026-03-26) がリリースされた。今回の目玉は **Protected Resource Metadata (PRM) エンドポイント**と **`enable-api` フラグ**だ。派手な破壊的変更はないが、エージェントツールの**認可モデル**に対する考え方が根本的に変わる可能性を秘めている。

## Protected Resource Metadata：OAuth 2.0 の進化形

### PRM とは何か

Protected Resource Metadata は、[RFC 9728](https://datatracker.ietf.org/doc/html/rfc9728) で定義された OAuth 2.0 の拡張仕様だ。リソースサーバー（この場合は GenAI Toolbox）が自身の保護リソースに関するメタデータを公開するエンドポイントを提供する。

従来の OAuth 2.0 フローでは、クライアントは認可サーバーの情報を事前に知っている必要があった。PRM はこれを逆転させ、**リソースサーバー自身が「自分にアクセスするにはどの認可サーバーを使うべきか」を宣言**する。

### MCP エコシステムとの接続

MCP 2025-11-25 仕様で強化された **OIDC / エンタープライズ管理認可**と PRM は直接的な関係がある。MCP サーバーが PRM エンドポイントを持つことで、MCP クライアント（エージェント）が認可サーバーを**自動発見**できるようになる。

MCP の未リリース変更として記録されている **RFC 8414 Well-Known URI 仕様**も同じ方向性だ。RFC 8414（認可サーバーメタデータ）と RFC 9728（保護リソースメタデータ）は補完関係にあり、両方が実装されると **認可フローの完全自動化**が可能になる。

| 仕様 | 役割 | GenAI Toolbox での実装 |
|------|------|----------------------|
| RFC 8414 | 認可サーバーの自動発見 | MCP 側で対応予定 |
| RFC 9728 | リソースサーバーの認可要件宣言 | v0.31.0 で PRM エンドポイント追加 |

### Manual PRM オーバーライド

v0.31.0 では PRM エンドポイントの自動発見に加えて、**Manual PRM オーバーライド**もサポートされている。これは以下のシナリオで重要だ：

- **開発環境**: 本番の認可サーバーを使わずにローカルでテストしたい
- **マルチテナント環境**: テナントごとに異なる認可サーバーを指定したい
- **レガシー統合**: RFC 9728 非対応の認可サーバーとの接続

オーバーライドの存在は、PRM 自動発見が**デフォルト動作**であることを意味する。「自動が基本、手動は例外」という設計思想だ。

## `enable-api` フラグ：攻撃対象面の明示的管理

### フラグの設計思想

`enable-api` フラグは、GenAI Toolbox の API エンドポイントを明示的に有効化するためのフラグだ。デフォルトでは API が無効化されていると推測される（API を使う場合は明示的にフラグを立てる）。

これは ADK Python v1.27.4 の**ビルダーエンドポイント web フラグゲート化**と同じ設計パターンだ。「使わない機能は公開しない」という最小権限原則を、フラグレベルで実装している。

### v0.30.0 の `serve` サブコマンドとの関係

v0.30.0 で追加された `serve` サブコマンドは、GenAI Toolbox をサーバーとして起動するコマンドだ。`enable-api` フラグはこの `serve` コマンドのオプションとして機能すると考えられる。

```
# API を有効化してサーブ（推測）
genai-toolbox serve --enable-api

# API なしでサーブ（MCP のみ）
genai-toolbox serve
```

MCP 経由のツール提供と HTTP API 経由のツール提供を**選択可能**にすることで、デプロイメントの柔軟性とセキュリティの両立を図っている。

## Dataplex ルックアップコンテキストツール：データディスカバリの深化

### コンテキスト検索の意味

v0.30.0 で追加された Dataplex `search_entries` スコープに続き、v0.31.0 では**ルックアップコンテキストツール**が追加された。search（検索）と lookup（参照）の違いは重要だ：

| 操作 | 用途 | 入力 |
|------|------|------|
| search_entries | キーワードベースのデータ探索 | 検索クエリ |
| lookup context | 既知エンティティのコンテキスト取得 | エンティティ ID |

lookup は「このテーブルはどのデータパイプラインの一部か」「このカラムの上流データソースは何か」といった、**データリネージとコンテキストの取得**に使われる。エージェントがデータについて「知る」ための深い情報源だ。

### ADK Python v1.28.0 の BigQuery 1P 移行との連携

ADK Python v1.28.0 の BigQuery 1P Toolset 移行と、GenAI Toolbox v0.31.0 の Dataplex ルックアップコンテキストは、**データ理解の二層構造**を形成する：

1. **データアクセス層**: ADK Python の BigQuery 1P Toolset でデータにアクセス
2. **データ理解層**: GenAI Toolbox の Dataplex ツールでデータのコンテキストを理解

エージェントが「データを操作する」だけでなく「データを理解する」ことができるようになる。

## BigQuery impersonation と UUID ベース分離

### impersonation の安全性

**BigQuery impersonation** は、エージェントがユーザーの代わりに BigQuery にアクセスする際の認証パターンだ。Service Account の impersonation を使用することで、エージェント自身の権限ではなくユーザーの権限でクエリを実行できる。

これは v0.28.0 で追加された **BigQuery 用カスタム OAuth ヘッダー名サポート**の延長線上にある。認証の柔軟性を段階的に拡大している。

### UUID ベース分離

**UUID ベースのアイソレーション**は、マルチテナント環境でのツール実行を安全に分離する機能だ。各テナント（またはリクエスト）に UUID を割り当て、ツール実行のコンテキストを分離する。

v0.30.0 の **One Skill per Toolset** と組み合わせると、「1スキル = 1ツールセット = UUID で分離されたコンテキスト」という明確な境界が生まれる。

## v0.27.0 からの 5 リリースを俯瞰する

v0.27.0 から v0.31.0 までの 5 リリースを振り返ると、一貫したテーマが見える：

| バージョン | テーマ | 破壊的変更 |
|-----------|--------|-----------|
| v0.27.0 | 設定の近代化 | 設定ファイル v2 |
| v0.28.0 | データソース拡大 | なし |
| v0.29.0 | 構造の再編 | ツールセット再構成、テレメトリ |
| v0.30.0 | スキル化 | なし |
| v0.31.0 | 認可と安全性 | なし |

**「構造化 → 拡大 → 再編 → 標準化 → 安全化」**という進化パターンだ。v0.29.0 の破壊的再構成で土台を作り、v0.30.0 でスキル標準化、v0.31.0 で認可モデルを整備した。

## 未リリースの展望

v0.31.0 以降、以下の開発が進行中だ：

- **Native MCP AlloyDB / HTTP**: MCP ネイティブのデータベースアクセス
- **Looker Agent**: Looker エージェント統合
- **API 統合リファクタ**: API の統合作業

Native MCP の方向性は、GenAI Toolbox が「MCP サーバーとしてのツール提供」を第一級の配信チャネルとして位置づけていることを示す。`enable-api` フラグによる HTTP API の選択的有効化と合わせると、**MCP がデフォルト、HTTP API がオプション**という優先順位が見えてくる。

## まとめ：認可の自動化がエージェントツールの鍵

v0.31.0 の変更は地味に見えるが、PRM エンドポイントの追加は**エージェントツール連携の認可自動化**への重要な一歩だ。エージェントが「どのツールにアクセスするか」を決める際に、認可要件を自動発見できることは、**ゼロコンフィグのツール連携**への道を開く。

MCP エコシステム全体（RFC 8414 + RFC 9728）の認可自動化と、GenAI Toolbox 固有の `enable-api` フラグによる攻撃対象面管理。この二層のセキュリティ設計が、v0.31.0 の本質だ。
