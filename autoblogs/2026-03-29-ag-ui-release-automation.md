# AG-UI release/2026-03-28 — リリース自動化とADKミドルウェアO(1)最適化が示すプロトコルの成熟段階

AG-UI プロトコルが release/2026-03-28 をリリースした。変更内容は 3 つ：**リリース自動化パイプライン統合**、**Dojo オートスケーリング修正**、**ADK ミドルウェア O(1) ルックアップ最適化**。一見すると機能追加はない。しかし、この「機能追加がないリリース」こそが、AG-UI がプロトコルとして成熟段階に入ったことを示している。

## リリース自動化：手動リリースからの卒業

### なぜ今リリース自動化なのか

AG-UI のタグ履歴を見ると、これまでのリリースは手動で行われていたと推測される。`ag-ui-agent-spec@0.1.0`、`ag_ui_adk@0.5.2`、`ag_ui_langroid@0.1.0`、`ag-ui-claude-sdk@0.1.0` など、**パッケージごとに異なる命名規則**のタグが混在していた。

リリース自動化の導入は、以下を意味する：

| 変更前 | 変更後 |
|--------|--------|
| 手動タグ付け | 自動リリースパイプライン |
| パッケージごとの個別リリース | 統合リリースプロセス |
| 不定期リリース | 自動化による安定サイクル |
| 命名規則の揺れ | 統一的な命名規則（推測） |

### MCP エコシステムとの対比

MCP プロジェクトは日付ベースのバージョニング（2025-11-25, 2025-06-18 等）を採用しており、リリースプロセスは比較的確立されている。MCP-Apps も v1.3.2 まで安定したリリースサイクルを持つ。

一方、AG-UI はこれまで v0.0.x 台のバージョンで急速に機能追加を重ねてきた。v0.0.48 の段階でリリース自動化を導入するのは、**v0.1.0（または v1.0.0）への道筋を整備する布石**と読める。

### A2A v1.0.0 のリリースプロセスとの並行

A2A は v1.0.0 のリリースに際して、Extension ガバナンスドキュメントの追加や SDK 後方互換性設計など、**リリースインフラの整備**を同時に行った。AG-UI のリリース自動化は、A2A が v1.0.0 に至る過程で踏んだステップを追体験しているとも言える。

## ADK ミドルウェア O(1) ルックアップ最適化

### 何が最適化されたか

AG-UI の ADK（Agent Development Kit）ミドルウェアにおけるルックアップ処理が **O(1) 時間計算量**に最適化された。ミドルウェアのルックアップとは、受信したメッセージやイベントを適切なハンドラーにルーティングする処理だ。

従来は（推測だが）リニアサーチ O(n) またはそれに近い計算量だったと考えられる。エージェントの数やイベントタイプが増加するにつれてパフォーマンスが劣化する問題があったのだろう。

### O(1) 化の実装パターン

一般的に、ルーティングテーブルの O(1) 化は以下のいずれかで実現される：

| 手法 | 特徴 | 適用シナリオ |
|------|------|------------|
| ハッシュマップ | キーベースの直接参照 | イベントタイプ → ハンドラーのマッピング |
| 配列インデックス | 整数インデックスでの直接アクセス | Enum ベースのイベントディスパッチ |
| コンパイル時マッチング | ビルド時にルーティングを確定 | 静的なプロトコルハンドラー |

AG-UI の場合、A2A v1.0.0 で導入された **SCREAMING_SNAKE_CASE Enum** フォーマットとの整合性を考えると、Enum ベースのハッシュマップが有力だ。

### パフォーマンスへの実際の影響

AG-UI プロトコルは以下のイベントタイプを処理する：

- テキストストリーミングイベント（TEXT_MESSAGE_START, TEXT_MESSAGE_CONTENT, TEXT_MESSAGE_END 等）
- ツールコールイベント（TOOL_CALL_START, TOOL_CALL_ARGS 等）
- 推論イベント（REASONING_START, REASONING_CONTENT 等）
- 状態管理イベント（STATE_DELTA, STATE_SNAPSHOT 等）

v0.0.48 で追加された**並列 LRO ツールコール**のような、複数のイベントが同時に発生するシナリオでは、ルックアップの計算量が直接的にレイテンシに影響する。O(1) 化は並列処理の性能改善に寄与する。

## Dojo オートスケーリング修正

### Dojo とは

Dojo は AG-UI のデモ・テスト環境として機能するバックエンドサービスだ。v0.0.48 で導入された Dojo バックエンドサービスとデプロイ設定（render.yaml）の延長線上にある。

オートスケーリングの修正は、Dojo が**実運用に近い負荷テスト環境**として使われていることを示す。エージェントの数がスケールした際の挙動を Dojo で検証し、AG-UI プロトコル自体の耐久性をテストしている。

### リリース自動化との関連

リリース自動化の導入と同時にオートスケーリング修正が入ったのは偶然ではないだろう。自動化されたリリースパイプラインには自動テストが含まれ、Dojo 環境でのスケーリングテストがその一部を担うと推測される。

## AG-UI の SDK 多様化と成熟度マトリクス

release/2026-03-28 のタグ履歴から、AG-UI が以下の SDK をサポートしていることがわかる：

| SDK | バージョン | 成熟度 |
|-----|-----------|--------|
| ag-ui-agent-spec | 0.1.0 | 初期 |
| ag_ui_adk | 0.5.2 | 開発中 |
| ag_ui_langroid | 0.1.0 | 初期 |
| ag-ui-claude-sdk | 0.1.0 | 初期 |

加えて、v0.0.48 までに以下のフレームワーク統合が追加されている：

- **LangGraph** (Python/TypeScript v0.0.28)
- **Mastra**
- **CrewAI**
- **Strands.py** (v0.1.2)
- **Ruby SDK** (#865)
- **Dart SDK** (v0.1.0)
- **Kotlin SDK** (v0.2.7)

SDK の多様化は、プロトコルの安定性が求められる段階に入ったことを意味する。SDK ごとに互換性を維持する必要があり、リリース自動化はその負担を軽減するための**インフラ投資**だ。

## 他プロトコルとの成熟度比較

| プロトコル | 現行バージョン | リリース自動化 | 破壊的変更頻度 | 成熟段階 |
|-----------|--------------|--------------|--------------|---------|
| A2A | v1.0.0 | あり | 低（v1 安定） | 成熟 |
| MCP | 2025-11-25 | あり | 低 | 成熟 |
| AG-UI | release/2026-03-28 | **今回導入** | 中（安定化中） | **移行中** |
| A2UI | v0.9 | なし | 中 | 開発中 |
| UCP | v2026-01-23 | なし | 高 | 開発中 |
| ACP | v2026-01-30 | なし | 高 | 開発中 |

AG-UI は「開発中」から「成熟」への**移行段階**にある。リリース自動化の導入は、この移行の明確なマイルストーンだ。

## まとめ：機能追加がないリリースの価値

release/2026-03-28 は「新機能がないリリース」だが、それ自体がメッセージだ：

1. **リリース自動化** = プロセスの成熟。手動作業からの脱却
2. **O(1) 最適化** = パフォーマンスの基盤整備。機能追加が来る前の地ならし
3. **Dojo スケーリング修正** = テストインフラの安定化。品質保証の基盤

「機能を追加しない」選択ができるのは、プロトコルが十分な機能セットを持ち、**品質とプロセスに投資する余裕がある**からだ。AG-UI は次の段階——おそらく v0.1.0 またはそれ以降の安定版リリース——への準備を始めている。
