# ADK Python v1.28.0 — A2A lifespan・Spanner Admin・マルチターンメトリクスが示すエージェント運用基盤の成熟

v1.27.4 のセキュリティ強化リリースからわずか 2 日後、ADK Python v1.28.0 (2026-03-26) が登場した。セキュリティパッチの後に機能爆発リリースが来るのは v1.27.0 → v1.27.3 のパターンと同じだが、v1.28.0 のテーマは明確に**エージェント運用基盤の成熟**だ。A2A lifespan パラメータ、Spanner Admin Toolset、MCP サンプリングコールバック、マルチターンタスクメトリクス——これらが描くのは「エージェントを作る」から「エージェントを運用する」への転換だ。

## A2A lifespan パラメータ：プロトコル間接続の寿命管理

### 何が変わったか

A2A v1.0.0 の正式リリースと同期して、ADK Python に **A2A lifespan パラメータ**のサポートが追加された。これはエージェント間通信の接続寿命を明示的に管理する機能だ。

A2A v1.0.0 で導入された `tasks/list` メソッド（フィルタリング・ページネーション付き）との組み合わせで、エージェント間のタスク委譲が「投げっぱなし」から「寿命管理付き委譲」に進化する。

### 設計上の含意

lifespan パラメータの導入は、エージェントが長時間実行されるシナリオを正式にサポートすることを意味する。従来の A2A 通信は短命なリクエスト-レスポンスが前提だったが、lifespan 管理により以下のパターンが可能になる：

| パターン | lifespan | ユースケース |
|---------|----------|------------|
| Fire-and-forget | 短い | 単純なツール呼び出し委譲 |
| Monitored delegation | 中程度 | 進捗追跡付きタスク委譲 |
| Long-running collaboration | 長い | マルチエージェント協調作業 |

ADK Go v1.0.0 以降で開発中の **Progressive SSE ストリーミング**と合わせて考えると、Go と Python の両方で長時間実行エージェントへの対応が進んでいることがわかる。

## BigQuery 1P Toolset 移行：ファーストパーティの意味

### 1P（First-Party）への移行

**BigQuery 1P Toolset 移行**は、サードパーティのラッパーやコミュニティ実装から Google 公式のファーストパーティ統合への移行を指す。v1.27.0 で追加された **BigQuery Dataplex Catalog 検索ツール**や **BigQuery プラグイン強化**（fork safety, auto views, trace continuity）の延長線上にある。

この移行が重要なのは、GenAI Toolbox v0.29.0 での **プリビルトツールセット再構成**（BigQuery を含む破壊的変更）と並行して進んでいる点だ。ADK Python 側は 1P 統合に移行し、GenAI Toolbox 側はツールセット構造を再編する——**二つのプロジェクトが同時にBigQuery 統合の品質を引き上げている**。

### GenAI Toolbox v0.31.0 との関連

GenAI Toolbox v0.31.0 で追加された **BigQuery impersonation** 機能は、ADK Python の 1P 移行と補完関係にある。impersonation（なりすまし認証）は、エージェントが特定のユーザー権限でBigQuery にアクセスするパターンを安全に実現する。

## Spanner Admin Toolset：データベース管理のエージェント化

### なぜ Spanner Admin なのか

Spanner は Google Cloud の分散データベースで、強整合性と水平スケーラビリティを両立する。**Spanner Admin Toolset** の追加は、データベースの**管理操作**（スキーマ変更、インスタンス管理、バックアップ等）をエージェントが実行できることを意味する。

v1.27.0 で追加された **Bigtable クラスタメタデータツール**と合わせると、Google Cloud のデータストア管理全般がエージェントツールとしてカバーされつつある。

| Toolset | 対象 | 操作レベル |
|---------|------|-----------|
| BigQuery 1P Toolset | データ分析 | クエリ・データ操作 |
| Spanner Admin Toolset | 分散DB | 管理操作 |
| Bigtable クラスタメタデータ | NoSQL | メタデータ参照 |
| GCS Skills (v1.27.0) | ファイルストレージ | 読み書き |

### 運用自動化への布石

Spanner Admin ツールがエージェントに開放されたことで、以下のような運用自動化シナリオが現実的になる：

- **インシデント対応エージェント**: Spanner のパフォーマンス問題を検知し、自動的にインスタンス設定を調整
- **スキーマ管理エージェント**: DDL 変更のレビューと適用を自動化
- **コスト最適化エージェント**: 使用パターン分析に基づくインスタンスサイジングの提案

## MCP サンプリングコールバック：プロトコル境界の拡張

### サンプリングコールバックとは

MCP（Model Context Protocol）のサンプリング機能は、MCP サーバーがクライアント側の LLM にリクエストを送信する**逆方向の通信パターン**だ。通常の MCP フローは「クライアント → サーバー（ツール実行）」だが、サンプリングは「サーバー → クライアント（LLM 呼び出し依頼）」を実現する。

ADK Python にサンプリングコールバックのサポートが追加されたことで、MCP サーバーが ADK エージェントの LLM 能力を借りて推論を実行できるようになる。

### エコシステム的な位置づけ

MCP 2025-11-25 仕様で安定した Extensions フレームワーク (SEP-2133) と組み合わせると、サンプリングコールバックは **MCP サーバーのインテリジェンス化**を加速する。GenAI Toolbox v0.31.0 の **Protected Resource Metadata エンドポイント**と合わせて、MCP 経由のツール連携がより高度になる。

## マルチターンタスクメトリクス：エージェント品質の定量化

### 3 つのメトリクス

v1.28.0 で追加された 3 つのマルチターンタスクメトリクスは、エージェントの評価を根本的に変える：

| メトリクス | 測定対象 | 意味 |
|-----------|---------|------|
| **success** | タスク完了率 | エージェントが目標を達成したか |
| **trajectory** | 実行経路 | 目標達成までの経路の効率性 |
| **tool trajectory** | ツール使用経路 | ツール選択の適切さ |

従来の単一ステップ評価（正解/不正解）から、**経路の品質**を評価できるようになった。

### v1.26.0 の User Personas との連携

v1.26.0 で評価フレームワークに導入された **User Personas** は、異なるユーザーペルソナでエージェントの応答品質を評価する機能だった。マルチターンメトリクスと組み合わせると、「ペルソナ A では trajectory が効率的だがペルソナ B では非効率」といった、**ペルソナ別の経路品質分析**が可能になる。

### ADK Go v1.0.0 の Replay プラグインとの補完

ADK Go v0.6.0 で追加された **Replay プラグイン** (#618) は、実際のエージェント実行をキャプチャして再生する機能だ。Python 側のマルチターンメトリクスと Go 側のリプレイ機能は補完関係にある——Go でキャプチャした実行を Python のメトリクスで評価する、というクロス言語評価パイプラインが理論上可能になる。

## Slack 統合と SSE ストリーミング：エージェントの接続先拡大

### Slack 統合

ADK Python に **Slack 統合**が追加された。これは ADK エージェントが Slack をインターフェースとして利用できることを意味する。AG-UI プロトコルの各種 SDK 統合（Claude Agent SDK, Langroid, Dojo）とは異なるアプローチで、ADK は**プラットフォーム直接統合**を選択している。

### 適合テスト向け SSE ストリーミング

**SSE ストリーミング**の適合テストサポートは、ADK Go v1.0.0 以降で開発中の **Progressive SSE ストリーミング**との適合性を確保するためだ。Go と Python で SSE ストリーミングの挙動が一致することは、A2A マルチ言語エコシステムの信頼性に直結する。

## v2.0.0a1 アルファが並走する意味

v1.28.0 のリリースと同時期に **v2.0.0a1 アルファ版**がタグ付けされている。v1.x 系に大量の新機能が投入され続ける一方で v2 の開発が進行するという構図は、v1.x 系の**安定性と機能性の両立**が限界に近づいていることを示唆する。

v1.28.0 で追加された機能群——特に A2A lifespan、MCP サンプリングコールバック、マルチターンメトリクス——は、v2.0.0 で**破壊的に再設計**される可能性のある領域だ。v1.x での実装は v2 の設計を検証するための**プロダクション実験**と見ることもできる。

## まとめ：「作る」から「運用する」への転換

v1.28.0 の変更を俯瞰すると、テーマは明確だ：

- **A2A lifespan**: エージェント間通信の寿命管理 = **運用**
- **BigQuery 1P 移行**: 公式統合への移行 = **品質保証**
- **Spanner Admin Toolset**: データベース管理のエージェント化 = **運用自動化**
- **MCP サンプリングコールバック**: プロトコル間連携の深化 = **統合強化**
- **マルチターンメトリクス**: エージェント品質の定量化 = **運用監視**

ADK Python は「エージェントを構築するフレームワーク」から「エージェントを構築・運用・監視するプラットフォーム」への進化の途上にある。v2.0.0 がこの方向性をどう結晶化させるかが、次の注目点だ。
