# MCP-Apps v1.3.1 — PDF 機能強化とモバイル対応が示す MCP アプリケーション層の成熟

2026-03-23 にリリースされた MCP-Apps v1.3.1 は、PDF 機能の大幅強化、モバイルサポート改善、重複プラグイン削除、そして detached-buffer バグ修正を含む。MCP (Model Context Protocol) エコシステムの中で、MCP-Apps が担う **アプリケーション層**としての設計思想と、v1.1.x から v1.3.x への進化軌跡を分析する。

## v1.3.x の主要変更——4つの柱

MCP-Apps v1.3.1 の変更は 4 つの柱に整理できる：

| 柱 | 変更 | 設計的意味 |
|----|------|-----------|
| PDF 強化 | アノテーション、interact ツール、ページ抽出 | LLM の文書処理能力の拡張 |
| 重複削除 | 重複 pdf-server プラグイン削除 | コードベースの整理・正規化 |
| モバイル対応 | map, PDF, threejs サンプルのモバイル対応 | マルチデバイス展開の実用化 |
| バグ修正 | detached-buffer バグ修正、古い E2E アサーション修正 | 本番運用品質の向上 |

### PDF Server 強化の詳細

v1.3.x の **PDF Server 強化** は、単なる機能追加ではなくアーキテクチャの変化を意味する。

追加された機能：
- **アノテーション**: PDF ドキュメントへの注釈追加
- **interact ツール**: PDF とのインタラクティブな操作
- **ページ抽出**: 特定ページの抽出と処理

これらは v1.1.x で入った **堅牢なパスバリデーション**、**フォルダ/ファイルルートサポート**、**HTTPS 必須化** の上に積み上がっている。v1.1.x でセキュリティ基盤を固め、v1.3.x で機能を充実させる——これは ADK Python の「セキュリティ → 機能」サイクルと同じパターンだ。

### 重複 pdf-server プラグイン削除の意味

「重複した pdf-server プラグインを削除（正規コピーは knowledge-work-plugins に）」という変更は小さく見えるが、**プラグインエコシステムのガバナンス** を示している。

MCP-Apps は複数のプラグインで構成される。知識作業系のプラグイン（knowledge-work-plugins）と汎用サンプルの分離は、**関心事の分離**という設計思想の現れだ。PDF 処理は「知識作業」の中核であり、そのプラグインを知識作業系に正規化することは、将来の機能拡張の方向性を示している。

## v1.2.x の基盤——Zod v4 対応と API 拡張

v1.3.x の前の v1.2.x では、**Zod v4 互換性修正** (#548) と **ListServerResources / readServerResource API** 追加が行われた。

Zod v4 対応は ADK Python の依存関係管理と対応する。v1.27.4 で LiteLLM 脆弱バージョンを除外したように、MCP-Apps も依存ライブラリの互換性維持を重要視している。

| v1.2.x の変更 | PR/Issue | 詳細 |
|--------------|---------|------|
| Zod v4 互換性修正 | #548 | `zod/v4` からのインポート修正 |
| モバイルサポート改善 | #555 | map, PDF, threejs サンプル対応 |
| react-with-deps バンドル修正 | #539 | SDK+zod のバンドル修正 |
| E2E セキュリティテスト改善 | #540 | frames[] トラバーサルによるテスト修正 |
| ListServerResources API | - | 新規 API 追加 |
| readServerResource API | - | 新規 API 追加 |
| テーマ対応ロゴ | #545 | `picture` 要素によるロゴ |

**ListServerResources / readServerResource API** の追加は特に重要だ。これは MCP サーバーが保持するリソースをクライアントから列挙・読み取りできる機能で、**LLM がどのリソースにアクセスできるかを動的に発見できる** 仕組みだ。

## detached-buffer バグ修正の技術的意味

v1.3.x で修正された **detached-buffer バグ**（range re-request 時）は、ストリーミングデータ処理の深部に潜む問題だ。

「detached buffer」とは、ArrayBuffer が他のオブジェクトに所有権を移譲（transfer）した後に、元のバッファへのアクセスを試みた際に発生するエラーだ。range re-request（部分的なデータの再取得リクエスト）はストリーミング PDF ビューアに典型的なユースケースであり、ページをスクロールするたびにバイト範囲指定でデータを取得する実装で発生しうる。

このバグが本番運用で顕在化したことは、**モバイルデバイスや低速ネットワーク環境での PDF 閲覧** が実際のユースケースとして成立してきたことを意味している。理論上の機能ではなく、実際に使われているからこそバグが発見・修正される。

## MCP-Apps の位置づけ——MCP エコシステムの中で

### MCP 仕様との関係

MCP 本体の現行バージョン 2025-11-25 は、以下のセキュリティ関連変更を含んでいる：

- **MCP Apps CSP**: アンバンドルアセット向け Content Security Policy 要件文書化
- **SSRF セキュリティ**: SSRF 対策ドキュメント追加
- **SECURITY.md 追加** (#472): GitHub Security Advisories ガイダンス

MCP-Apps はこれらの仕様要件に対応した実装例（リファレンスアプリ）としての位置づけを持つ。MCP-Apps v1.1.x での **HTTPS 必須化** と **SECURITY.md 追加** は、MCP 仕様の CSP・SSRF 対策との整合だ。

### ADK Python との深い結合

ADK Python v1.27.0 で追加された **MCP App UI Widgets**（MCPTool で MCP App UI ウィジェットサポート）と **MCPB パッケージング**（pdf-server 用 Claude Code プラグイン）により、MCP-Apps と ADK Python は深く統合されている。

この統合の意味：

```
MCP-Apps v1.3.1 (PDF アノテーション + interact)
    |
    +-- ADK Python v1.27.0 (MCP App UI Widgets)
    |       |
    |       +-- LlmAgent が MCP-Apps の UI を生成
    |       +-- MCPTool が MCP-Apps と通信
    |
    +-- Claude Code プラグイン (MCPB パッケージング)
```

Legend / 凡例:
- MCP-Apps: MCP プロトコル上に構築されたアプリケーション層
- ADK Python: Google の Agent Development Kit
- MCPTool: ADK が MCP サーバーと通信するためのツール抽象

PDF アノテーションと interact ツールが追加されたことで、LLM エージェントが PDF ドキュメントと双方向にやり取りできるシナリオが広がった。

## モバイルサポートの戦略的意味

MCP-Apps が map、PDF、threejs サンプルのモバイル対応を行ったことは、**MCP の展開領域の拡大** を示している。

現在の MCP Apps サポートクライアントは Claude Desktop、ChatGPT、MCPJam、Postman、VSCode、goose、Copilot と多岐にわたる。これらの一部はモバイルデバイスからアクセスされるケースもある。特に：

- **map サンプル**: モバイルでの位置情報利用
- **PDF サンプル**: スマートフォンでのドキュメント閲覧
- **threejs サンプル**: 3D ビジュアライゼーションのモバイル展開

モバイル対応の技術的難しさは、タッチイベントの処理、画面サイズの適応、パフォーマンス最適化にある。v1.2.x と v1.3.x の両方でモバイルサポートが言及されているのは、この課題が複数の反復で改善されてきたことを示している。

## v1.1.x から v1.3.1 への変化の軌跡

| バージョン | 主な変更 | 方向性 |
|-----------|---------|--------|
| v1.1.x | downloadFile, PDF パスバリデーション, HTTPS 必須化, SECURITY.md (#472) | セキュリティ基盤確立 |
| v1.2.x | Zod v4 (#548), モバイル (#555), E2E (#540), ListServerResources API | 互換性と API 拡張 |
| v1.3.1 | PDF 強化（アノテーション・interact・ページ抽出）, 重複削除, モバイル, detached-buffer 修正 | 機能充実と品質向上 |

この軌跡は **「基盤 → 互換性 → 機能充実」** という成熟パターンを整理すると明確だ。セキュリティ基盤（v1.1.x）を固めた後、依存関係互換性と API 拡張（v1.2.x）を行い、本格的な機能充実（v1.3.x）へと進んでいる。

## A2UI との連携——未来の方向性

changelogs.md の未リリース変更には **MCP Apps 統合: A2UI に MCP Apps を統合** (#748) と **McpApps A2UI コンポーネント: MCP Apps 用コンポーネント導入** (#801) が挙げられている。

A2UI v0.9 の **Prompt First 設計**（構造化出力優先からプロンプト埋め込み優先への哲学変更）と MCP-Apps の統合は、**LLM がユーザーインターフェースを動的に生成しながら MCP サーバーと通信する** という次世代のエージェント UI パターンを示している。

MCP-Apps v1.3.1 の PDF 機能強化（アノテーション、interact、ページ抽出）は、この A2UI 統合への伏線だ。LLM が PDF にアノテーションを付けながら A2UI の `updateComponents` メッセージでリアルタイムに UI を更新するシナリオが技術的に実現可能になる。

## まとめ

MCP-Apps v1.3.1 は、PDF 機能強化・重複削除・モバイル対応・バグ修正という 4 つの柱で **MCP アプリケーション層の実用段階への成熟** を示している。v1.1.x でのセキュリティ基盤確立から始まり、v1.2.x での互換性改善、v1.3.x での機能充実——この軌跡は MCP エコシステムが実験的なプロトコルから本格的な開発プラットフォームへと進化していることを意味する。

ADK Python v1.27.0 との深い統合（MCP App UI Widgets、MCPB パッケージング）と、A2UI v0.9 との統合計画（#748、#801）を合わせると、MCP-Apps は単独のアプリケーションではなく **マルチエージェント・マルチインターフェース・マルチプロトコルのエコシステムの接合点** として進化する方向性が見える。2026-03-23 のリリースは、その進化の一里塚だ。
