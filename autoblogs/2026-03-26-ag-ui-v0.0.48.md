# AG-UI v0.0.48 — 初バージョンタグとセッション設計の進化が示すプロトコル成熟の方向性

AG-UI (Agent-User Interaction Protocol) が 2026-03-24 に v0.0.48 をリリースした。この**初の正式バージョンタグ**付きリリースは、AG-UI が実験的プロトコルから安定した実装基盤へと移行したことを示している。connectAgent() ハングバグ修正、並列 LRO ツールコール対応、セッションキャッシュのキー設計変更——これら 3 つの変更が持つ設計的意味と、AG-UI のアーキテクチャ上の位置づけを整理する。

## v0.0.48 が「初バージョンタグ」である意味

AG-UI は以前、バージョンタグなしの継続的デプロイで開発が進められていた。v0.0.48 での初タグ付けは、**API の一定の安定性が確保された** ことを示す重要なシグナルだ。

v0.0.x という慎重な番号付けは意図的だ。AG-UI は A2A v1.0.0（2026-03-12 正式リリース）、A2UI v0.9、MCP 2025-11-25 といった複数のプロトコルと接触点を持つ。これらが安定化したタイミングで、AG-UI 自身のバージョン管理を開始することには整合性がある。

## connectAgent() ハングバグ修正——非同期設計の修正

### バグの性質

「`connect()` が同期的にスローした場合にハングする問題」は、**非同期/同期の境界での例外伝播** という古典的な問題だ。

AG-UI の `connectAgent()` は、エージェントバックエンドとの接続を確立するコア機能だ。通常の非同期エラー（Promise rejection など）は正しく処理されるが、同期的な例外（接続設定の検証エラーなど）が接続フロー内で発生した場合にハングするケースがあった。

このバグは開発体験（DX）に直結する。エージェント接続が応答なくハングすると、開発者はタイムアウトまで待つか、プロセスを強制終了するしかない。修正後は同期例外が適切に伝播し、デバッグ可能な状態になる。

### 設計的な考察

このバグが v0.0.48 で修正されたことは、AG-UI が **幅広いエージェントバックエンドとの統合テスト** を経たことを示唆している。AG-UI が統合する SDK/フレームワーク一覧：

| フレームワーク | 統合バージョン |
|-------------|--------------|
| Python SDK | v0.1.14 |
| Strands.py | v0.1.2 |
| LangGraph | langgraph python/typescript v0.0.28 |
| Langroid | Langroid フレームワーク統合 |
| Claude Agent SDK | #916 |
| Bedrock Converse API | LangGraphAgent での互換性修正 (#1300) |
| Kotlin SDK | v0.2.7 (#1103) |
| Dart SDK | v0.1.0 |
| Ruby SDK | フルプロトコル実装 (#865) |

これだけ多様なバックエンドと統合するプロトコルでは、同期/非同期境界の問題は特定の実装でのみ顕在化する。**多様な実装との統合が進んだからこそ発見・修正できたバグ** だ。

## 並列 LRO ツールコール——エージェント UI の体験設計

### LRO とは何か

LRO (Long-Running Operations) は、完了まで時間がかかるツール呼び出しを指す。データベースクエリ、外部 API 呼び出し、機械学習推論——エージェントが使うツールの多くは非同期で時間がかかる。

AG-UI v0.0.48 の **並列 LRO ツールコール**（ADK での並列 Long-Running Operations ツールコールハンドリング）は、複数の LRO を同時に実行しながらその状態を UI にストリーミングする。

### 設計上の重要性

並列 LRO がなければ、エージェントは複数のツールを **順次** 実行するしかない。例えば「気象データを取得しながら地図を生成する」タスクでは：

| 方式 | 所要時間 | ユーザー体験 |
|------|---------|------------|
| 順次実行 | 気象(2秒) + 地図(3秒) = 5秒 | 画面がフリーズ |
| 並列 LRO | max(気象, 地図) = 3秒 | 各ツールの進捗が随時更新 |

ADK Go v1.0.0 以降の未リリース変更に **Progressive SSE ストリーミングと並列関数実行** があることと、AG-UI v0.0.48 の並列 LRO ハンドリングは **バックエンドとフロントエンドプロトコルの設計整合** を示している。バックエンド（ADK Go の並列関数実行）とフロントエンドプロトコル（AG-UI の並列 LRO 表示）が同時期に進化している。

### JSON Schema メタフィールド除去との関連

v0.0.48 の変更に **JSON Schema メタフィールド除去**（バリデーション前のメタフィールドストリッピング）がある。LRO ツールコールの並列実行では、ツールの入出力スキーマの厳密なバリデーションが重要になる。メタフィールドがバリデーション前に除去されることで、スキーマの純粋なバリデーションが可能になる——これは並列 LRO 対応の補完的な変更だ。

## セッションキャッシュの (thread_id, user_id) キーイング——セキュリティと設計

### クロスユーザー衝突問題

旧来のセッションキャッシュが `thread_id` のみでキーイングされていた場合、異なるユーザーが同じ `thread_id` を持つと **セッションデータが衝突** する可能性がある。マルチユーザー環境では、UUID などの一意な ID でも衝突確率はゼロではない。

`(thread_id, user_id)` の複合キーによるキーイングは、**ユーザー識別とスレッド識別の両方を保証** する。これは単なるバグ修正ではなく、**マルチユーザー対応の設計変更** だ。

### ADK Python との設計的対応

ADK Python v1.27.0 で追加された **AuthProviderRegistry**（`CredentialManager` 内のプラガブル認証統合）と、この `user_id` によるセッションキー付けは設計的に対応する。認証された `user_id` でセッションをキーイングすることで、認証システムとセッション管理が整合する。

AG-UI v0.0.48 のセッションキャッシュ改善は、**セキュリティと正確性の両方を満たす設計への移行** だ。

## LANGSMITH_API_KEY 環境変数サポート

v0.0.48 の **Render Service 改善**（`LANGSMITH_API_KEY` 環境変数サポート追加）は小さな変更だが、**LangSmith との公式統合** を意味する。

LangSmith は LangChain/LangGraph の可観測性プラットフォームだ。AG-UI が LangGraph との統合を持ち（langgraph python/typescript v0.0.28）、LangSmith のトレーシングをサポートすることで、**LangGraph ベースのエージェントを AG-UI プロトコルで接続した場合のデバッグ体験** が向上する。

AG-UI、LangGraph、LangSmith の三角関係：

```
LangSmith (可観測性)
    |
    | LANGSMITH_API_KEY
    v
AG-UI Render Service
    |
    | AG-UI Protocol
    v
LangGraph Agent
```

Legend / 凡例:

- LangSmith: LangChain エコシステムの可観測性ツール
- AG-UI Render Service: AG-UI プロトコルのレンダリングサービス
- LangGraph Agent: LangGraph で実装されたエージェントバックエンド

AG-UI は多数の SDK とフレームワークを統合しながら、v0.0.48 で初のタグ付きリリースを行った。以前の主要変更を整理すると：

| 変更 | Issue/PR | 意味 |
|------|---------|------|
| Claude Agent SDK 統合 | #916 | Anthropic エコシステムとの正式接続 |
| Bedrock Converse API 互換性修正 | #1300 | AWS エコシステムとの互換性 |
| REASONING イベント移行 | - | `THINKING_*` → `REASONING_*` へのリネーム |
| A2UI チャット統合 | #1177 | A2UI プロトコルとの統合 |
| Reasoning Spec | #1050 | 推論仕様の追加 |

| NX 移行 | #1092 | Turbo から NX へのビルドシステム移行 |

これだけの変更を経てのバージョンタグ付けは、**多様な統合を経た成熟の宣言** だ。

## AG-UI の設計哲学——イベント駆動プロトコルの成熟

AG-UI の設計思想は **イベント駆動のストリーミングプロトコル** だ。エージェントの状態変化（REASONING の開始/終了、ツール呼び出し、テキスト生成）をイベントとしてクライアントにストリーミングする。

v0.0.48 の並列 LRO ハンドリングは、このイベント駆動設計の自然な進化だ。単一のツールコールをストリーミングするだけでなく、**複数の並列ツールコールを個別のイベントストリームとして管理**できることで、複雑なエージェント動作のリアルタイム表示が可能になる。

JSON Patch デルタ（RFC 6902）サポートと組み合わせると、「状態の差分をリアルタイムで更新しながら、複数の並列ツールが動作する」という高度なエージェント UI が実現できる。

## まとめ

AG-UI v0.0.48 の初バージョンタグ付きリリースは、**AG-UI プロトコルのプロダクション利用への準備完了** を宣言するものだ。connectAgent() ハングバグ修正（非同期/同期境界の修正）、並列 LRO ツールコール対応（エージェント UI の体験向上）、セッションキャッシュの `(thread_id, user_id)` キーイング（マルチユーザー安全性）——これら 3 つの変更は、それぞれが独立した価値を持ちながら、**エンタープライズ利用に耐えるプロトコル** への方向性を示している。

ADK Go v1.0.0 の Progressive SSE と並列関数実行、ADK Python v1.27.0 の AuthProviderRegistry、A2UI v0.9 の Prompt First 設計——これらと AG-UI v0.0.48 の変更が整合的に進化していることが、エージェント UI エコシステム全体の成熟を示している。2026-03-24 は、その成熟の一里塚となる日付だ。
