# ADK Python v1.27.3-v1.27.4 セキュリティ強化——任意モジュールインポート防止と LiteLLM 脆弱バージョン排除の設計思想

2026-03-24 前後にリリースされた ADK Python v1.27.3 と v1.27.4 は、機能追加ではなく **セキュリティ強化** を主眼とするリリースだ。任意モジュールインポート防止（v1.27.3）と LiteLLM 脆弱バージョン除外（v1.27.4）——これら 2 つの変更が示す**セキュリティ設計思想の進化**と、v2.0.0a1 アルファが並走する ADK Python エコシステムの現在地を分析する。

## v1.27.3：任意モジュールインポート防止の背景

### 攻撃ベクターの解剖

「任意モジュールインポート防止」という変更は、一見地味だが **エージェントセキュリティの核心** に触れる。Python の動的インポート機能（`importlib.import_module()` 等）をエージェントが悪意のある入力から実行した場合、攻撃者がシステムライブラリや秘密情報へのアクセスを得られる可能性がある。

ADK Python では、エージェントがツールを通じてコードを実行するシナリオが増加している。v1.27.0 で追加された **BashTool**（SkillToolset への追加）や **GCS Skills** は、エージェントが外部コードを実行する機会を増やす。これらの機能が充実するほど、任意モジュールインポートのリスクは高まる。

この変更が v1.27.0（機能爆発）の 3 バージョン後に来たことは、**「機能追加 → セキュリティ強化」の開発サイクル**を示している。

### Python 固有の動的インポートリスク

ADK Go v1.0.0 や ADK JS v0.2.4 には、この種のモジュールインポート防止変更は存在しない。Go は静的型付け・コンパイル時依存解決であり、実行時の任意モジュールロードは言語設計上困難だ。

Python 固有のリスクと対策：

| リスク | 詳細 | 対策 |
|--------|------|------|
| `importlib.import_module()` の悪用 | プロンプトインジェクション経由でのシステムライブラリアクセス | モジュール名のホワイトリスト化 |
| 動的インポート直接呼び出し | エージェントが生成したコード内でのインポート | コード実行サンドボックスの強化 |
| 文字列評価経由 | 動的なコード実行 | 実行前の AST 解析 |
| 不正なデシリアライゼーション | 信頼できないデータのデコード | 安全なシリアライゼーション形式の使用 |

### v1.27.2 からの連続性

v1.27.2 では **Dataplex OAuth スコープ修正**（BigQueryToolset の OAuth スコープ修正）と **usage_metadata ハンドリング改善**（Vertex AI での usage_metadata 処理）が入った。認証スコープの精密化はセキュリティ強化の一環であり、**OAuth スコープ精密化 → モジュールインポート防止** という セキュリティ改善の流れが読める。

## v1.27.4：LiteLLM 脆弱バージョン除外の戦略的意味

### 依存関係セキュリティの現代的アプローチ

v1.27.4 の **LiteLLM 脆弱バージョン除外** は、依存関係管理の観点で注目すべき変更だ。「危険な LiteLLM バージョンを除外する」という措置は、サプライチェーンセキュリティへの対応を示している。

LiteLLM は ADK Python が複数のモデルプロバイダーを統一 API で扱うための抽象レイヤーとして使用している。v1.27.0 で **LiteLLM 出力スキーマ**（LiteLLM モデルでツールと出力スキーマの同時使用対応）が追加されたことで、LiteLLM への依存深度が増している。

依存関係の深度と脆弱性リスクは比例する。LiteLLM の特定バージョンに脆弱性が発見された場合、ADK Python ユーザーが自動的に安全なバージョン範囲に誘導される仕組みを構築したことになる。

### ビルダーエンドポイントの Web フラグゲート化

v1.27.4 のもう一つの変更、**ビルダーエンドポイントを web フラグでゲート化**（ビルダーエンドポイント制限）は、**攻撃対象面の縮小**という設計思想を体現する。

Web エンドポイントとして公開されるビルダー機能は、ネットワーク経由のアクセスを前提としている。フラグによるゲート化は「使わない機能は公開しない」という最小権限原則の適用だ。

ADK Go v1.0.0 では **pathPrefix 設定** (#616) によるエンドポイント制御が v0.6.0 で入っており、Go と Python で **エンドポイントアクセス制御の強化** という方向性が共有されている。

## v1.27.1 のロールバックが示す開発プロセスの成熟

v1.27.3 と v1.27.4 の間には v1.27.1 のロールバックという出来事がある。**LlmAgent 修正のロールバック**（バージョンフィールド欠落による LlmAgent 作成問題の修正を戻した）は、一度リリースした修正に問題が見つかり迅速にロールバックした事例だ。

これは **ADK Python の開発プロセスの成熟** を示している。問題のある修正を素早く検出してロールバックし、v1.27.2 でより安定した修正を提供する——この速度と柔軟性はエージェントフレームワークの信頼性基盤だ。

## v2.0.0a1 との関係——セキュリティ設計の継承

v2.0.0a1 アルファは、2026-03-24 以前にタグ付けされた。v1.27.3（任意モジュールインポート防止）と v1.27.4（LiteLLM 脆弱バージョン除外）の直後に v2.0.0a1 が存在することは、**v1.x のセキュリティ強化を v2 に継承する** 設計意図を示唆している。

v2.0.0a1 で予想されるセキュリティ関連の変化：

| 領域 | v1.x での対応 | v2.0.0 での方向性 |
|------|--------------|-----------------|
| モジュールインポート | v1.27.3 で防止（patch） | サンドボックス化の標準化 |
| 依存関係管理 | v1.27.4 で脆弱バージョン除外 | 依存関係スキャンの自動化 |
| OAuth スコープ | v1.27.2 で精密化 | 最小権限原則のデフォルト適用 |
| エンドポイント制御 | v1.27.4 で web フラグゲート | アクセス制御の宣言的設定 |

## ADK エコシステム全体のセキュリティ成熟度

v1.27.3 と v1.27.4 の変更を、ADK エコシステム全体のセキュリティ成熟という文脈で置くと：

### ADK JS の未リリース変更との比較

ADK JS v0.2.4 以降の未リリース変更には **パストラバーサル防止**（FileArtifactService のパストラバーサル脆弱性修正、CWE-22）が含まれている。Python の任意モジュールインポート防止と同じく、**ファイルシステムアクセス制御の強化**という方向性が 2 言語で並走している。

### MCP-Apps との連携セキュリティ

MCP-Apps v1.3.1（2026-03-23）では **PDF Server 強化**（堅牢なパスバリデーション、HTTPS 必須化）と **SECURITY.md 追加**（GitHub Security Advisories ガイダンス、#472）が過去バージョンで入っている。ADK Python の v1.27.3 セキュリティ強化は MCP ツールを通じた攻撃ベクターへの対応でもある。

ADK Python v1.27.0 で追加された **MCP App UI Widgets**（MCPTool で MCP App UI ウィジェットサポート）は、ADK と MCP-Apps を深く結合させた。この結合深度が増すほど、MCP-Apps 側のセキュリティ強化も ADK セキュリティの一部となる。

## 破壊的変更の回避戦略

v1.27.3 の任意モジュールインポート防止は、**破壊的変更を引き起こす可能性がある修正** だ。既存のエージェントが動的インポートに依存していた場合、この修正によって動作が変わりうる。

しかし ADK Python の破壊的変更履歴を見ると：

| バージョン | 変更 |
|-----------|------|
| v1.24.0 | `credential_manager` が `tool_context` を受け取るよう変更 |
| v1.23.0 | OpenTelemetry BigQuery プラグイントレーシング移行 |

これら以外での破壊的変更は最小限に抑えられている。セキュリティ強化でも **後方互換性を最大限維持** しながら対策を講じる——これが v1.x 系の一貫した方針だ。

## ADK Python v1.27.x のバージョン変遷

| バージョン | リリース時期 | 主な変更 |
|-----------|------------|---------|
| v1.27.0 | 2026-03-17 以前 | 機能大幅追加（BashTool, GCS Skills, A2A, UiWidget） |
| v1.27.1 | 2026-03-17 前後 | LlmAgent 修正（その後ロールバック） |
| v1.27.2 | 2026-03-20 前後 | Dataplex OAuth スコープ修正、usage_metadata 改善 |
| v1.27.3 | 2026-03-22 前後 | 任意モジュールインポート防止（セキュリティ） |
| v1.27.4 | 2026-03-24 | LiteLLM 脆弱バージョン除外、ビルダーエンドポイントゲート化 |

この変化の流れは **機能拡張 → 安定化 → セキュリティ強化** という ADK Python の成熟パターンを整理すると明確だ。v1.27.0 の機能爆発後、v1.27.1 のロールバックを挟んで v1.27.2 で修正、v1.27.3-v1.27.4 でセキュリティ強化という一連の流れが実用段階のフレームワーク開発らしい進化を示している。

```python
# ADK Python v1.27.3 以降: 安全なモジュール利用の設計指針
# 動的インポートの代わりに、明示的な依存宣言を推奨
from google.adk.tools import BashTool  # 明示的インポート
from google.adk.agents import LlmAgent

agent = LlmAgent(
    model="gemini-2.0-flash",
    tools=[BashTool()],
)
```

## まとめ

ADK Python v1.27.3 の任意モジュールインポート防止と v1.27.4 の LiteLLM 脆弱バージョン除外は、**エージェントフレームワークのセキュリティ成熟** という文脈で読むべき変更だ。BashTool・GCS Skills・MCP App UI Widgets という「エージェントが外部と接触する機会を増やす」機能を v1.27.0 で大量投入した後、v1.27.3-v1.27.4 でそのセキュリティ基盤を固める——この「拡張 → 強化」のサイクルが ADK Python の設計思想を示している。

v2.0.0a1 が並走するタイミングでのセキュリティ強化は、**v2 系に継承されるべき安全な基盤を v1 系で確立する**という意図が読める。ADK Go v1.0.0（Go 言語の静的安全性）、ADK JS v0.2.4（未リリースの CWE-22 修正）と合わせて、3 言語の ADK エコシステムが各言語の特性に応じたセキュリティ設計を進化させていることが意味している。
